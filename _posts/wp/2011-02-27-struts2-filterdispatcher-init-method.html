很久没有看SSH了，今天看看struts2的源码，想知道其框架的实现和运行原理，加深自己对struts2的理解。
分析struts2的初始化过程，需要建立一个工程，然后对其进行调试、跟踪代码的运行过程。
新建一个helloworld工程，可以参照这篇文章：【为Struts 2.0做好准备】，算是为后面的工作做好准备吧。

在web.xml里可以看到struts2的启动入口是org.apache.struts2.dispatcher.FilterDispatcher这个类，该类实现了StrutsStatics, Filter两个接口。Filter是servlet的过滤器接口，有三个抽象方法：init()、doFilter()、destroy()。StrutsStatics借口定义了struts的action context或是其他接口使用的一些常量。

熟悉Filter的话，就会知道FilterDispatcher类的初始化主要是在init()方法里进行的。

该方法如下： 

   public void init(FilterConfig filterConfig) throws ServletException {
        try {
	    //1.设置filterConfig
            this.filterConfig = filterConfig;

//2.记录日志
initLogging();

//3.创建转发器
dispatcher = createDispatcher(filterConfig);
//4.创建转发器
dispatcher.init();
//5.注入该dispatcher
dispatcher.getContainer().inject(this);

//5.注入该dispatcher
staticResourceLoader.setHostConfig(new FilterHostConfig(filterConfig));
} finally {
ActionContext.setContext(null);
}
}

该方法用到了几个成员变量： 

    //提供一个日志记录的实例.
    private Logger log;

//提供一个action映射的实例, 该实例通过set方法注入
private ActionMapper actionMapper;

//一个封装Filter参数的实例,该类来自servlet, 在init方法里进行set设置.
private FilterConfig filterConfig;

//暴露一个转发器实例给子类.
protected Dispatcher dispatcher;

//加载静态资源，该实例通过set方法注入
protected StaticContentLoader staticResourceLoader;

init方法执行步骤： 1.设置filterConfig。
2.记录日志。该类先从Filter过滤参数中读取key为loggerFactory的值，然后通过该值反射生成一个日志的工厂类，将这个生成的工厂类注入到LoggerFactory类中。
从这点可以知道struts2支持用户自定义一个日志工厂类。 

 private void initLogging() {
        String factoryName = filterConfig.getInitParameter("loggerFactory");
        if (factoryName != null) {
            try {
                Class cls = ClassLoaderUtils.loadClass(factoryName, this.getClass());
                LoggerFactory fac = (LoggerFactory) cls.newInstance();
                LoggerFactory.setLoggerFactory(fac);
            } catch (InstantiationException e) {
                System.err.println("Unable to instantiate logger factory: " 
			+ factoryName + ", using default");
                e.printStackTrace();
            } catch (IllegalAccessException e) {
                System.err.println("Unable to access logger factory: " 
			+ factoryName + ", using default");
                e.printStackTrace();
            } catch (ClassNotFoundException e) {
                System.err.println("Unable to locate logger factory class: " 
			+ factoryName + ", using default");
                e.printStackTrace();
            }
        }

log = LoggerFactory.getLogger(FilterDispatcher.class);

}

3.创建一个Dispatcher，代码如下，该方法可以被子类覆盖。该类的主要作用就是从 filterConfig获得初始化参数，然后封装成一个HashMap，获得ServletContext，然后创建一个Dispatcher对象，将其返回。 

protected Dispatcher createDispatcher(FilterConfig filterConfig) {
        Map< string , String> params = new HashMap< string , String>();
        for (Enumeration e = filterConfig.getInitParameterNames(); e.hasMoreElements();) {
            String name = (String) e.nextElement();
            String value = filterConfig.getInitParameter(name);
            params.put(name, value);
        }
        return new Dispatcher(filterConfig.getServletContext(), params);
}

4.Dispatcher是什么类？主要作用是什么？从api中可以知道，该类是一个工具类，实际的转发器委托大部分的代码给这个工具类实现。每一个原生的转发器都拥有一个该类的实例，供所有请求共享。

Dispatcher最为关键的init方法，该方法主要是为了加载配置文件，包括xml的和零配置的配置，并且跟新一些可选的配置参数，如编码、开发模式、是否加载资源文件等等。 

public void init() {
    	if (configurationManager == null) {
    	   configurationManager = new ConfigurationManager(BeanSelectionProvider.DEFAULT_BEAN_NAME);
    	}

try {
init_DefaultProperties(); // [1]
init_TraditionalXmlConfigurations(); // [2]
init_LegacyStrutsProperties(); // [3]
init_CustomConfigurationProviders(); // [5]
init_FilterInitParameters() ; // [6]
init_AliasStandardObjects() ; // [7]

Container container = init_PreloadConfiguration();
container.inject(this);
init_CheckConfigurationReloading(container);
init_CheckWebLogicWorkaround(container);

if (!dispatcherListeners.isEmpty()) {
for (DispatcherListener l : dispatcherListeners) {
l.dispatcherInitialized(this);
}
}
} catch (Exception ex) {
if (LOG.isErrorEnabled())
LOG.error("Dispatcher initialization failed", ex);
throw new StrutsException(ex);
}
}

关于加载配置文件，xwork2定义了很多Provider，例如：
[1]中的DefaultPropertiesProvider，用于加载Properties文件
[2]中的XmlConfigurationProvider和StrutsXmlConfigurationProvider，用于加载xwork.xml或是struts的xml文件。
[3]中的LegacyPropertiesConfigurationProvider
[5]中的ConfigurationProvider，用于加载自定义的配置文件。
[6]中的匿名内部类的ConfigurationProvider，用于加载Filter中初始化参数。
[7]中的BeanSelectionProvider

Dispatcher类中还有cleanup()和init_PreloadConfiguration()方法，涉及到ConfigurationProvider和injection的相关类。

总结，struts把读取配置文件的工作分给了不同的provider来处理，并引入了inject的思想，这两点之后在深入研究。
